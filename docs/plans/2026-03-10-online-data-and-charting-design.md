# Phase 14: Online Data Layer, Dual-Mode Charting & Drawing Tools

Version: 1.0
Date: 2026-03-10
Status: **Approved**

---

## Problem Statement

Fiboki production runs on a 2.3MB starter dataset (7 forex majors, H1 only) bundled in the Docker image. The full canonical dataset (961MB, 60 instruments × 6 timeframes = 360 parquet files) exists locally but is not accessible in production. This means charts show limited data, backtests only run for 7 instruments at one timeframe, and research cannot execute the full matrix. The platform must become fully online.

Additionally, charts currently display only static historical data with no live price feed, no interactive drawing tools, and no drawing persistence.

---

## Target Architecture

### 1. Online Historical Data — Railway Persistent Volume

**Decision:** Mount a Railway persistent volume at `/data` on the backend service.

**Why this over alternatives:**
- Object storage (S3/R2) adds latency, new service dependency, and credential management for ~1GB of data that fits trivially on a volume
- Expanding the Docker image to 1GB+ makes every redeploy slow and wastes bandwidth
- Railway volumes are $0.25/GB/month (~$0.25 total), single-region matches our single backend, and reads are local filesystem speed

**Volume structure** (mirrors local dev exactly):

```
/data/
  canonical/
    manifest.json          ← generated index of all datasets
    histdata/
      eurusd/
        eurusd_h1.parquet
        eurusd_m1.parquet
        eurusd_m5.parquet
        eurusd_m15.parquet
        eurusd_m30.parquet
        eurusd_h4.parquet
      gbpusd/
        ...
    dukascopy/
      ...
  starter/                 ← Docker-bundled fallback only
    histdata/
      eurusd/eurusd_h1.parquet
      ...
  fixtures/                ← legacy, transitional
```

**Path resolution:** `FIBOKEI_DATA_DIR=/data` set as Railway env var. Existing `get_data_root()` in `paths.py` already respects this. No code change needed for path resolution.

**Load order** (unchanged):
1. `canonical/dukascopy/` (validation-grade)
2. `canonical/histdata/` (bulk research)
3. `starter/histdata/` (Docker-bundled fallback)
4. `fixtures/` (legacy)

When the volume has the full dataset, steps 1–2 resolve immediately. Starter only serves as fallback if the volume is unmounted or a file is missing.

**Observable degradation:** When the backend falls back from canonical to starter or fixtures, it logs a warning with the instrument/timeframe and the fallback source. The `GET /api/v1/system/status` response includes a `data_source` field (`"volume"`, `"starter"`, `"fixtures"`) indicating the active data root.

### 2. Data Manifest

A `manifest.json` at the root of the canonical directory, generated by a CLI command:

```json
{
  "version": 1,
  "generated_at": "2026-03-10T12:00:00Z",
  "generator": "fibokei manifest generate",
  "datasets": [
    {
      "symbol": "EURUSD",
      "provider": "histdata",
      "timeframe": "H1",
      "bars": 17520,
      "from_date": "2023-01-02",
      "to_date": "2024-12-31",
      "size_bytes": 412800,
      "path": "histdata/eurusd/eurusd_h1.parquet",
      "modified_at": "2026-03-10T12:00:00Z",
      "checksum": "sha256:abc123..."
    }
  ]
}
```

**API endpoint:** `GET /api/v1/data/manifest` — serves the manifest as JSON. Cached in memory on first read, invalidated by `POST /api/v1/data/manifest/refresh`.

**Frontend uses:** The instruments endpoint derives `has_canonical_data` dynamically from the manifest (true if at least one dataset exists for that symbol). The frontend can show available timeframes per instrument before users attempt operations.

### 3. Market Data Endpoint Improvements

**Query parameters added to `GET /api/v1/market-data/{instrument}/{timeframe}`:**

| Param | Type | Default | Constraint |
|-------|------|---------|------------|
| `from` | ISO datetime | (none) | Optional start bound |
| `to` | ISO datetime | (none) | Optional end bound |
| `limit` | int | 2000 | Max 10,000 |
| `mode` | string | `historical` | `historical` or `live` (future) |

**Server-side caps:** Hard max 10,000 bars per response. If no `from`/`to` given, returns the last `limit` bars. Response includes metadata: `total_bars`, `from_date`, `to_date`, `source` (which provider/path served the data).

**Performance:**
- Replace `iterrows()` with vectorized `df.to_dict('records')` — 10–50x faster
- LRU cache for loaded DataFrames: max 50 entries, 5-minute TTL. Key = `(symbol, timeframe, provider)`. Cache is process-local (no external dependency).

### 4. Dual Chart Mode Architecture (Future Slice)

**Historical mode** (this slice): canonical parquet datasets. Full date range browsing.

**Live mode** (future slice): IG demo/live REST API via server-side polling.
- Backend `IGDataProvider` wraps existing `IGClient.get_prices()`, normalizes to OHLCV
- New bars cached in memory, served via same endpoint with `?mode=live&limit=200`
- Frontend SWR hook adds `refreshInterval: 5000` when mode is live
- Ichimoku precomputed server-side in both modes (hard rule)
- Same `MarketDataResponse` contract for both modes — frontend is mode-agnostic

**Frontend switching:** Mode toggle in `ChartToolbar`. Changes `mode` param in the SWR key.

### 5. Drawing Tools Architecture (Future Slice)

**Built-in klinecharts overlays** (zero custom rendering required):

| Tool | klinecharts name |
|------|-----------------|
| Trendline | `segment` |
| Extended line | `straightLine` |
| Ray | `rayLine` |
| Horizontal line | `horizontalStraightLine` |
| Price level | `priceLine` |
| Fibonacci retracement | `fibonacciLine` |
| Parallel channel | `parallelStraightLine` |
| Annotation | `simpleAnnotation` |

**Custom overlay needed:** Rectangle / supply-demand zone (~30 lines using `rect` figure).

**Persistence:** Database table `chart_drawings` with columns: `id`, `user_id`, `instrument`, `timeframe`, `tool_name`, `points` (JSONB), `styles` (JSONB), `created_at`, `updated_at`.

**API:**
- `GET /api/v1/charts/drawings?instrument=&timeframe=`
- `POST /api/v1/charts/drawings`
- `PUT /api/v1/charts/drawings/{id}`
- `DELETE /api/v1/charts/drawings/{id}`

On chart mount: load drawings → replay via `chart.createOverlay({ name, points })`.

### 6. Backtests / Research / Optimisation

**What the Railway volume unlocks:**
- All 60 instruments × 6 timeframes available for backtests
- Research matrix can run the full grid (720 combos per strategy)
- Advanced research (walk-forward, Monte Carlo, validation) all use `load_canonical()` — they just work

**Endpoint changes:**
- Remove dead `data_path` parameter from backtest schema
- Research frontend uses manifest to show available combos
- Instruments endpoint derives `has_canonical_data` from manifest

**Performance guards:**
- Request timeout: 5 minutes for research/backtest runs
- Research matrix: max 100 combos per request (configurable)

## Operator Workflow

### Initial volume population

```bash
# 1. Create Railway volume (via dashboard or CLI)
railway volume create fiboki-data --size 2GB

# 2. Mount to backend service at /data
# (Railway dashboard: Service → Volumes → Mount Path: /data)

# 3. Set env var
railway variables set FIBOKEI_DATA_DIR=/data

# 4. Upload canonical data to volume
# Option A: railway shell + rsync from local
railway shell
# then from local: rsync -avz data/canonical/ railway-host:/data/canonical/

# Option B: SCP via Railway's SSH access
scp -r data/canonical/ <railway-ssh>:/data/canonical/

# 5. Generate manifest on the server
railway run python -m fibokei manifest generate

# 6. Verify
curl https://api.fiboki.uk/api/v1/data/manifest | jq '.datasets | length'
# Expected: 360
```

### Subsequent data updates

```bash
# Upload new/updated parquet files
scp -r data/canonical/histdata/new_symbol/ <railway-ssh>:/data/canonical/histdata/new_symbol/

# Regenerate manifest
railway run python -m fibokei manifest generate

# Refresh cached manifest
curl -X POST https://api.fiboki.uk/api/v1/data/manifest/refresh
```
